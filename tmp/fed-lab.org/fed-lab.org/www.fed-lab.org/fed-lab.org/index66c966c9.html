Federation Lab

A test suite for identity federation protocols, OAuth, OpenID Connect and SAML

Use OPtest service

A service for testing OpenID providers are set up at https://op.certification.openid.net:60000/

Provider configuration

When entering the service the configuration page will be presented, see image 1

Image 1. Start page when visiting the configuration server

If a new configuration has been created you will need to enter some necessary information about the provider.First you need to enter how the application should fetch the provider information from the provider.

Dynamic discovery:

If the provider supplies the provider configuration choose dynamically and enter the issuer URL, see image 2.

Image 2. Dynamically supplied provider configuration

Static discovery:

If on the other hand the provider configuration is supplied statically you need to enter some required information, see image 3. You could add optional static provider metadata fields by clicking the button “Add static provider metadata field” and check the field you want to add.

Image 3 Statically supplied provider configuration

When all the provider configuration information is supplied click “Next” or “Client configuration”

Tets instance configuration

Every issuer could run up to five test instances at the same time. In order to simplify the management of the test instances the user could specify a test tool id. If there are existing instance id’s they will occur in a drop down list and could then be overwritten.

 

Image 4 The test tool instances management page.

Client configuration

In this configuration page you could specify how the test instance should behave.

Client registration

If the provider only supports static client registration a “client id” and “client secret” should be generated by the provider. While generating the client credentials enter the Redirect URI generated by the application. It is the URI to which the user should be redirected after completing a End-User authentication.

Select supported features:

Here all the features supported by the OpenID provider should be selected.

  • JWT signed with “None” algorithm (Unsigned)
  • JWT signed with algorithm other then “None”
  • Encrypted JWT
  • No alternative selected equals that JWT is not supported

Test specific request parameters:

Some tests are testing if the provider could handle specific request parameters. In order to run those tests you should specify the request parameters, see image 4 for example values.

Submit configuration:

In order to save the configuration on the server and start a test tool server click the submit button. When the configuration is stored on the server it’s possible to save configuration for later use by downloading it. When the configuration is submitted the application will ask whether an test tool instance should be started or not.

Image 5 Example of how to fill out client configuration

Test tool instance

The test tool consist of two part. First there is a current profile configurations, see image 5, where certain information could be modified in the profile configuration page. Secondly there are the test flows presented as a list, see image 6.

Current profile configurations:

The current configurations contains information on which response type the test tool is using, if the OpenID provider supports dynamic discovery or not, if dynamic client registration is supported or not and which cryptography features are supported. Both the response type and the cryptography features could be modified while running the test tool. It’s also possible to show extra tests

Image 6. Current profile configurations and a link to the profile configuration page

Test flows

Tests which matches the current profile is shown as a list and sorted by different categories. In order to run a test press the black button to the left of the test. Depending of the result the button should turn green while successful, yellow for warning or red if an error occurred. A complete list of all the legends could be found in Table 1.

Other than the result in form of a colour encoded button an information button should occur to the right of the test. When pressed it is possible to see the test output and a trace log which contains information about all the request made by the test tool and responses revived, see image 7 for an example.
